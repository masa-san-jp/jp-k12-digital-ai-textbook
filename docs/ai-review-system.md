# AI自律レビューシステム設計書（G2原稿審査の自律化）

**版:** 第0.1版（2026年7月18日・設計提案）
**位置づけ:** 仕様書第17章G2（原稿審査）を、人間の専門レビュアーを前提とせずに運用するためのシステム設計。本書は設計提案であり、採用には仕様書第17章・付録Fの改訂（本書第8章の改訂案文）を要する。採否はプロジェクト所有者の判断事項。

---

## 1. 設計原則

### 1.1 信頼の根拠を「人間であること」から「検証の性質」へ移す

人間の専門レビューが信頼される理由を分解すると、次の5性質に還元される。

| 性質 | 人間レビューでの実現 | 本設計での実現 |
|---|---|---|
| **独立性** | レビュアー≠執筆者 | 盲検レビュー（第4章）: レビュアーは原稿本文と判定基準のみを見る。執筆者の意図説明・セルフチェック記録は遮断 |
| **敵対性** | 専門家の批判的姿勢 | 全指摘に反証工程を必置（第5章）: 指摘は反証エージェントの攻撃に生き残って初めて確定 |
| **再現性** | 同一基準の適用（実際は個人差が大きい） | 同一プロンプト・同一基準・機械可読ログ。プロンプト版をハッシュで固定し、判定を再実行可能にする |
| **追跡可能性** | 判定記録（付録F） | 全指摘に基準ID＋原文引用＋提案修正を強制。引用のない指摘は無効（第6章） |
| **較正** | 資格・経験による暗黙の保証 | **変異注入テストによる検出率の実測**（第5章）。信頼を仮定せず測定する |

この分解に立つと、「人間でなければならない」根拠が実際に残るのは次の2点だけである。

1. **物理世界のデータ生成（G3実証授業）:** 実在の児童生徒の所要時間・達成率・つまずきは、教室で授業が行われて初めて存在するデータであり、生成できない。捏造は R9（時数確定プロトコル）と付録H の存在意義を破壊する。→ 本設計の対象外（ただし事前推定で不確実性を狭める。第7.1節）
2. **法的説明責任の署名（出版・検定時点）:** 有資格者のサインオフの価値は分析能力ではなく、誤りだった場合に責任を負う主体の存在にある。AIは分析を代替できるが、責任主体は代替できない。→ リポジトリ段階のゲートから出版判断時点の1回に集約する（第7.2節）

この2点以外——付録C 120項目・付録G 30項目・技術的正確性・統計語法・アクセシビリティ・発達適合の判定——はすべてテキストに対する基準参照判定であり、自律化の対象とする。

### 1.2 人間の関与は「労働」ではなく「例外処理と意思決定」に限定する

- 人間は行単位のレビュー作業を一切しない
- 人間に残るのは (a) 出版判断時の法的サインオフ（AIが証拠一式=dossierを整備済みの状態で受け取る）、(b) 裁定のデッドロック解消（発生時のみ）、(c) go/no-goの最終意思決定
- G3は「人間の労働」ではなく「現実世界の観測」であり、その分析・集計・R9判定は自律化する

### 1.3 本プロジェクトの前提との整合

READMEが宣言するとおり、本プロジェクトは「AIによる網羅性・一貫性・再現性の追求」を制作の前提としている。制作をAIが担いながら品質保証だけ人間の手作業を仮定するのは設計として非一貫であり、本書はQA層を制作層と同じ原理で再設計するものである。仕様書12.4（仮説検証の設計）が制作物に求める研究的アプローチを、レビューシステム自身にも適用する（＝検出率の実測）。

## 2. 検証可能性による4層アーキテクチャ

すべての判定基準を「どうやって真偽が確定するか」で4層に分類する。上の層ほど誤検出がなく安価で、下の層ほどLLM判定に依存するため敵対的検証と較正で固める。**層を混ぜないことが設計の要**——決定的に検査できる項目をLLMに判定させるのは無駄であり、逆は危険である。

### Tier 0: 決定的機械検査（CI常時実行・誤検出ほぼゼロ）

正規表現・辞書照合・構造解析で判定が確定する項目。`tools/validate_manuscripts.py`（新設）としてCIに組み込み、全push/PRで実行する。

| 検査 | 対応基準 |
|---|---|
| 1文の字数上限（小60字/中高80字） | C-016 |
| 統一用語辞書（付録E）未登録用語の検出・学年別言い換えの照合 | C-019, C-020 |
| 禁止表現（「絶対」「必ず安全」、AIの無注記擬人化、「失敗」をエラーの意で使用、訳語「幻覚」等） | C-026, C-027, 付録E注意欄 |
| 目標IDの書式・実在照合（goals.jsonとの突合） | C-003, C-006 |
| 9部構成テンプレートの節の欠落（小中）・とびら必須項目（時数・KC・接続欄） | C-007〜C-010 |
| アンプラグド代替の存在（小中はMUST） | C-012 |
| 差異化3ルート表・倫理討議・図版指示の代替テキスト列の存在 | C-013, C-038, C-095 |
| 二重否定・禁止動詞（付録B「理解する」「分かる」等）の目標文内使用 | C-023, C-001 |
| 原稿メタ情報の時数・紙面が執筆計画書2.3と一致 | 執筆計画書P1 |

### Tier 1: 計算検証（実行で真偽確定・誤検出ほぼゼロ）

数値・コード・論理は再計算・実行で機械的に検証する。

| 検査 | 方法 | 対応基準 |
|---|---|---|
| 本文中の全計算例（360÷6=60、70+85+60=215等） | 数式抽出→再計算 | C-052, C-065 |
| 擬似言語・コード例のトレース値・章末演習の正答 | 実装して実行、選択肢と突合 | C-051, C-057 |
| 単元時数合計＝巻総時数、ページ配分合計＝単元部予算 | 集計照合 | 執筆計画書2.4 |
| 前提参照の方向性（既存の`validate_goals.py`を原稿レベルへ拡張） | グラフ検証 | C-006, R3 |

### Tier 2: 基準参照LLM判定（意味判定・証拠引用強制）

付録C 120項目のうち、意味の理解を要するが判定基準が明文化されている項目（例: C-011「導入が操作説明から始まっていない」、C-035「比喩に限界注記がある」、C-082「ルーブリックB水準が目標文と論理一致」）。

- 判定は必ず**原文の該当箇所の引用**と**基準ID**を伴う。引用できない指摘は形式不備として自動棄却
- 1項目=1判定（「だいたい適合」を許さない3値判定: 適合/不適合/対象外）
- 不適合には提案修正を必須添付（修正不能なら「不能」と理由を明記）

### Tier 3: 専門的総合判定（開放的判断・パネル+定足数）

基準が明文化しきれない専門判断: 誤概念カタログの教育心理学的妥当性、発達段階との適合（この活動は小3に本当に可能か）、法規記述の通説整合、比喩が誘発しうる新たな誤概念の予見（C-054）等。

- 5分野レンズ（第3章）がそれぞれ独立に判定し、**定足数**（例: 重要度Aは2レンズ以上の一致）で確定
- Tier 3の指摘には確信度ラベル（高/中/低）を必須とし、低確信度は「要注意事項」として記録に残すのみ（差戻し事由にしない）

## 3. パネル構成（仕様書12.1の5分野に対応）

| ロール | 担当 | 数 |
|---|---|---|
| 分野レビュアー | AI/CS技術・統計・法務倫理・アクセシビリティ・児童発達の5レンズ。各原稿をTier 2〜3で判定 | 5/原稿 |
| 反証者（Verifier） | 各「不適合」指摘の反証を試みる。指摘とは独立のコンテキストで、原稿と指摘のみを受け取る | 1/指摘 |
| 裁定者（Arbiter） | レビュアーと反証者の判定が割れた指摘の裁定。両者の論拠のみで判断 | 発生時 |
| 修正者（Editor） | 確定指摘のうち機械的修正が可能なものを適用。修正後はTier 0/1を再実行 | 1/巻 |
| 完全性監査（Completeness Critic） | 「チェックリスト外の重大問題はないか」を巻単位で問う。チェックリスト過適合（項目は満たすが教育的に機能しない紙面）の検出が任務 | 1/巻 |

### 3.1 同一基盤モデル問題（本設計の最大の弱点と緩和策）

執筆もレビューも同系のLLMが担う場合、**盲点が相関する**（執筆時に見えなかった誤りはレビュー時も見えない可能性が高い）。これは人間の「レビュアー≠執筆者」が保証する誤り検出の独立性を、役割分離だけでは再現できないことを意味する。緩和策を4つ重ねる。

1. **盲検化:** レビュアーには原稿本文＋基準のみを渡す。執筆時のセルフチェック記録・コミットメッセージ・意図説明は見せない（自己申告へのアンカリング防止）
2. **敵対的役割設計:** 反証者は「指摘を潰す」側に、レビュアーは「見落としを恥じる」側にインセンティブを明示したプロンプトで非対称化し、誤りの出方を変える
3. **モデル多様性:** レビュアーと反証者に可能な限り異なるモデル（少なくとも異なる設定・独立コンテキスト）を割り当てる。較正（第5章）はモデルの組ごとに実施する
4. **非モデル依存の下支え:** Tier 0/1は決定的検査であり、モデルの盲点と無関係に機能する。相関盲点のリスクをTier 2/3だけに閉じ込める

なお原稿をレビュアーに渡す際は**データとして**渡す（原稿内に指示的文字列があってもレビュアーはそれを命令として実行しない規律をプロンプトで固定する）。

## 4. レビューの流れ（1巻あたり）

```
原稿 ─→ Tier 0 機械検査 ──不合格→ 修正者 ─→ 再検査
        ↓合格
        Tier 1 計算検証 ──不合格→ 修正者 ─→ 再検証
        ↓合格
        5分野レビュアー（並列・盲検）が Tier 2/3 判定
        ↓ 不適合指摘（基準ID＋引用＋提案修正）
        反証者が指摘ごとに反証を試行
        ↓ 生存指摘のみ            ↓ 反証成立
        （割れたら裁定者）          棄却（記録には残す）
        ↓ 確定指摘
        ├─ 機械的修正可 → 修正者が適用 → Tier 0/1 再実行
        ├─ 設計変更級（重要度A/B） → 差戻し（原稿の書き直し）
        ├─ G3データ待ち → g2-review-log に分類記録
        └─ 法的サインオフ待ち → dossier に集約（第7.2節）
        ↓
        完全性監査（巻単位）→ 検出事項は次ラウンドの入力
        ↓ 新規指摘が2ラウンド連続ゼロになるまで反復（loop-until-dry）
        G2判定記録（付録F互換・第6章）
```

差戻し判定は付録F.3の規則をそのまま使う（重要度A 1件で差戻し／B 3件以上で差戻し／Cは修正確認のみ）。

## 5. 較正プロトコル（本設計の核: 信頼を実測する）

パネルの判定を採用してよいかは、仮定ではなく**検出率の実測**で決める。

### 5.1 変異注入テスト（原稿へのミューテーションテスト）

1. 原稿のコピーに、既知の違反を計画的に注入する（例: 計算例を1桁誤らせる／比喩の限界注記を削る／ルーブリックB水準を目標文とずらす／目盛り不定のグラフ指示に差し替える／「絶対安全」を挿入する／小3原稿に付録Bで小3不可の動詞を入れる）。分野ごとに20件以上、重要度A/B/Cを混在させる
2. 注入位置を伏せたままパネルに全工程（レビュー→反証→確定）を実行させる
3. 分野別に **recall**（注入違反の検出率）と **precision**（確定指摘のうち真の違反の率）を算出する

### 5.2 合格基準と部分的自律化

| 指標 | 基準（初期値・運用で改訂） |
|---|---|
| 注入違反recall | 分野ごとに90%以上（重要度Aは100%） |
| 確定指摘precision | 80%以上（反証工程通過後） |

- **合格した分野のみ、その分野のG2判定権限をパネルに与える**。不合格の分野は判定を「参考情報」に降格し、当該分野に限り人間レビューを残す
- これが「人間を働かせない」への合理的な答えである: 実測で証明できた範囲を自律化し、証明できない範囲だけを人間に残す。範囲は較正の改善とともに縮小する
- モデル・プロンプト・基準のいずれかを変更したら再較正（プロンプト版ハッシュで変更を検知）

### 5.3 継続較正

G3実証で実際に発覚した紙面起因の問題（つまずき・誤概念の発生箇所）は「パネルが見逃した真の違反」として遡って記録し、recallの実地推定値を更新する。レビューシステム自身が第18章の改訂サイクルの対象となる。

## 6. 判定記録（付録F互換の機械可読ログ）

- 全判定を `quality-records/g2/<巻>.jsonl` に記録し、`docs/g2-review-log.md` を自動生成する
- 付録F.1の記載項目への対応: **判定者名 → ロール＋モデルID＋プロンプト版ハッシュ**、判定日、判定（合格/条件付/差戻し）、指摘（番号・重要度・基準ID・原文引用・内容・提案修正）、対応記録（修正コミットSHA・再判定結果）、エビデンス（Tier 0/1の検査ログ、較正実測値）
- 記録の完全性はCIで検査する（引用欠落・基準ID欠落の指摘はマージ不可）

## 7. 残余の人間関与の最小化設計

### 7.1 G3実証（物理世界の観測・代替不能）

代替はしないが、事前に不確実性を狭める自律工程を置く。

- **時数の事前推定:** 活動ステップ数×学年別標準所要（既知の授業研究の相場観）から単元所要時間を推定し、R9の暫定値±20%に収まる見込みを事前判定。危険域の単元はG3前に再設計
- **読み負荷の機械推定:** 学年配当漢字・文長・文構造から読み負荷を推定し、発達適合レンズの判定材料にする
- G3実施後のデータ分析・R9判定（3校中央値・±20%）・仕様書への時数反映は完全自律化する

### 7.2 法的サインオフ（説明責任の集約）

- リポジトリ段階のG2法務判定はAIパネル（法務レンズ）が担い、**出版・検定申請の判断時点に1回だけ**有資格者のサインオフを置く
- その時点で有資格者が受け取るのは、AIが整備済みの **dossier**: 付録G 30項目の判定＋根拠条文＋原文引用＋論点別のAI法的分析＋未確定論点（AI生成物の権利等）の明示。サインオフの作業量を「レビュー」から「監査」に圧縮する
- 教科書検定制度そのものは外部制約であり本設計の対象外

2. **12.2 G2通過基準:** 「専門レビュアー5分野の承認」→「5分野の専門レビュー機能の承認（AIパネルの場合は較正実測値・反証工程ログを含む判定記録を必置）」
3. **付録F.1 判定者欄:** 「判定者（氏名・役割）」→「判定者（氏名または AIロール＋モデルID＋プロンプト版ハッシュ・役割）」

## 9. 失敗モードと対策

| 失敗モード | 対策 |
|---|---|
| 相関盲点（執筆と同系モデルが同じ誤りを見逃す） | 3.1の4緩和策＋Tier 0/1の非モデル依存検査＋G3実地データによる継続較正（5.3） |
| ラバースタンプ化（全部「適合」で流す） | 変異注入テストが構造的に検出する（注入違反を見逃せばrecallが下がり判定権限を失う） |
| チェックリスト過適合（項目は満たすが教育的に死んだ紙面） | 完全性監査エージェントを基準外の観点で必置＋G3の達成率データで最終検証 |
| 誤検出の洪水（precision低下→修正コストで自壊） | 反証工程の必置＋引用強制＋precision基準（5.2）で確定指摘の質を担保 |
| 原稿内テキストによる指示注入 | 原稿はデータとして渡す規律をレビュアープロンプトに固定（3.1） |
| モデル・プロンプト変更による判定ドリフト | プロンプト版ハッシュ＋変更時の再較正義務（5.2） |

## 10. 実装計画（実行は別途の意思決定）

| 段階 | 内容 | 成果物 |
|---|---|---|
| S1 | Tier 0検査器の実装・CI組込み | `tools/validate_manuscripts.py`＋CI拡張 |
| S2 | Tier 1計算検証ハーネス | 数値抽出・再計算、擬似言語実行系 |
| S3 | 較正セット作成（変異注入・分野別20件以上）と1巻試走 | 較正用原稿群・recall/precision実測値 |
| S4 | 合格分野からパネル本運用（84文書×5レンズ＋反証、loop-until-dry） | `quality-records/g2/`・`docs/g2-review-log.md` |
| S5 | G3事前推定器（7.1）・dossier生成（7.2） | 時数推定レポート・法務dossier様式 |

規模感（S4フル実行時）: レビュー84×5＝420判定＋反証・裁定・監査で概算600〜800エージェント実行。段階導入（S3の1巻試走→較正合格→展開）により、較正不合格のまま全量を走らせる無駄を排除する。
